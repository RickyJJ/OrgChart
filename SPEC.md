# 📖 《青云志》项目规范 (SPEC)

## 1. 业务目标与验收标准

### 1.1 业务目标
- **产品名称**: 《青云志》 (The Path to Court)
- **核心目标**: 打造一个沉浸式的中国古代官制文化探索平台，科普古代官职的层级结构、实际职能、代表人物、历史典故与经典诗词。
- **目标用户**: 传统文化爱好者、历史系/文科学生、对古代职场及诗词典故感兴趣的大众用户。
- **MVP阶段目标**: 以**唐代**官职体系为切入点，完成核心的“官阶架构展示”、“诗词典故关联”以及基本的UI风格搭建。

### 1.2 验收体验标准 (UI/UX)
- **整体架构**: 左侧深色导航侧边栏 (Sidebar Navigation)，右侧为**无界宣纸画布主内容区 (Infinite Paper Canvas)**，取消硬界面的束缚。
- **整体风格**: 新中式极简风 (Neo-Chinese Minimalist) 与“水墨留白”结合。
- **品牌资产**: 适配专属品牌图标 `qyz_icon.png` 作为地址栏 Favicon；主 Logo `qyz_logo.png` (桌面端) 与专属移动端 Logo `qyz_logo_mobile.png`。
- **色彩规范**: 
  - 侧边栏及主字色：藏青/玄黑 (#1e293b, #1a1a1a)。
  - 主背景色：宣纸暖白底色 (#f5f5dc 变体)，叠加极微弱木浆纤维噪点。
  - 架构面板：采用 `bg-board-soft` (rgba(236, 232, 222, 0.92)) 配合 12px 毛玻璃基础模糊 (Backdrop Blur) 以提升沉浸感与可读性。
  - 强调色：朱砂红 (#af292e，用于印章、核心按钮、活动状态标签)。
- **排版与字体**: 标题和关键层级使用优雅的宋体 (Serif)，正文使用易读的黑体 (Sans-serif)。支持双语显示（中文主标题，英文副标题）。
- **细部控件特征**:
  - **根节点 (皇权)**: 使用高保真背景图 `emperor-bg.png`，且不显示标题文字和边框。
  - **朝代切换按钮**: 采用行楷字体排版。选中态背景为极具包裹感的专属印泥图片 (`switch-bg-1.png`)，点击时伴随专属“盖印 (Stamp)”原地位移缩放动效，文字始终维持深墨色。
  - **官职卡片**: 极简牌匾风格（尺寸由比例计算设定为 `60px * 167px`），强制使用纯中文竖版排版，字体设定为 `font-serif` (Noto Serif SC 1.6rem, Weight 600)。交互逻辑：
    - **PC 端**: 标题过长（超过 4 个字）时显示秒响应的悬停走马灯（Marquee）特效。
    - **移动端 (Touch)**: 点击选中卡片时，触发一次完整的走马灯滚动（动画时长自适应，匀速感），同时给予物理反馈（`.touch-feedback-active`：400ms 内小幅震动缩小并伴随 1.3x 亮度泛光增强，模拟“按压印章”的实感）。
  - **典故呼吸印记 (Lore Breathing Mark)**: 当接口数据表明该官职含有丰富的“典故连珠”数据（如关联著名历史人物或经典诗词）时，自动在卡片边缘点缀一枚极微小的朱砂红 (#af292e)  动态印章或水墨晕染点。该印记需自带极缓的呼吸动效（Opacity 0.6 ↔ 1.0 循环），作为“此处有故事”的隐性视觉引导（Affordance），激发用户探索欲。
  - **三明治式渲染层级**: 采用多层叠加结构——底层为 `card-bg-insight.png` 水墨底纹（带 Alpha 遮罩裁剪），中层为 `card-border-overlay` 墨灰色框线，顶层为文字层，通过 `background-size: 100% 100%` 实现零形变渲染。
  - **扇形展开动效 (Fan-out)**: 子节点收起时，父节点背后左右各以 `3deg` 角度旋转展开两张伪卡片，取代旧的横向堆叠方案，视觉更具古风韵律感。
  - **悬浮详情卡片**: 改为在右侧悬浮的立体卡片，包含全量英文翻译、引言关联列表等，消除主架构图的中英混排冗余。
- **核心动效**:
  - 朝代切换时的“双向缩放过渡”反馈。
  - 长名称官职悬停时的自动纵向走马灯（仅滚动一次）。
  - 初次加载时的“画卷平铺”入场动画 (Implemented).

### 1.3 移动端与响应式标准 (Mobile & Responsive UX)
- **核心布局**: PC 端布局保留，移动端 (`< 768px`) 侧边导航栏必须隐藏并退化为沉浸式底部导航（Bottom NavBar）或极简折叠菜单，释放横向极大化纵深给无界架构树。
- **触控手势**: 架构树必须原生支持“双指捏合缩放(Pinch-to-zoom)”与“单指滑动平移(Pan)”，禁用移动端默认的橡皮筋回弹或双击放大干扰 (`touch-action: none`)。
- **界面与抽屉降级**: 右侧详情面板在手机竖屏强制表现为自下而上的“上拉抽屉” (Bottom Sheet)，背景叠加深色半透明遮罩防止误触底部的树形图。
- **触屏替换逻辑**: 凡是桌面端依赖鼠标悬停 (Hover) 才能触发的信息（走马灯、数据 Tooltip 等）必须为其建立触碰按压 (Tap) 触发及其闭环（如点按外屏任意部位关闭）逻辑。
- **屏幕键盘与海报适配**: 表单区设计需要考虑手机唤出虚拟键盘对页面 `100dvh` 的坍塌。系统通过 html2canvas 合成的分享卡片必须在各类长宽比手机屏幕下做到无信息裁切。

### 2.2 官阶巡礼 (The Hierarchy Tree)
- **功能描述**: 多维组织架构图，直观展示三省六部等古代国家机器的运转模式。
- **交互与技术实现**:
  - 废除全局自适应缩放，采用“渐进式展开 + 视口跟随”的探照灯模式。初始锁定 `scale(1)` 且仅展示根节点和第一层级，由实际内容撑开画布大小。
  - 基于原生前端技术 (Vanilla JS + CSS) 和 `transform` 手势计算渲染交互式树状图并支持平滑拖拽/缩放 (Pan & Zoom)。
  - 支持节点点击排他性展开与折叠，点击父节点时自动平滑移动视界居中聚焦，确保 Noto Serif SC 可读性。
  - **组件卸载**: 对折叠的子树执行严格的 DOM 物理移除或设为 `display: none`，切勿仅使用 `opacity`，防止重排雪崩。含有未展开子节点时，在节点追加如 `[辖 24 司]` 的视觉线索。
  - **“云迹”面包屑导航 (Breadcrumbs)**: 
    - **结构与位置**: 抽离为独立 UI 组件，在主视图左上角与朝代标题集成。采用“朝代图标 + 路径序列”的横排流式排版。
    - **移动端智能折叠**: 在小屏幕或路径过长（>2节点且字数多）时，自动隐藏中间节点与根节点，展示为 `/ ... / 当前节点`。点击 `...` 弹出毛玻璃质感的下拉列表查阅全路径。
    - **实时焦点同步 (Focal Sync)**：建立“视界即路径”原则。只要官职卡片获得焦点、被手动展开、或通过搜索/平移使其居中，云迹将实时同步该节点的溯源路径。
    - **交互规则**: 点击云迹节点执行平滑归位（Pan-to-center）及路径补全展示，但**不会**主动触发官职详情面板的弹出，以保证纯净的架构游历体验。
- **MVP 范围**: 优先实现并完善“唐代”与“明代”中央官制的树状图（如尚书省、中枢省、内阁及下辖机构）。

### 2.3 双层信息架构与交互 (Two-Level Detail UX)
- **第一层 (内联卡片)**: 
  - 样式：白底水墨晕染卡片，包含中文名及英文翻译（如：中书省 Secretariat）。
  - 动作：点击选中卡片，触发边缘发光高亮，并呼出悬浮详情卡片。**支持重复点击切换（Toggle）开启/关闭状态。**
- **第二层 (悬浮详情卡片)**:
  - 触发：点击第一层卡片。
  - 内容：在屏幕右侧弹出的悬浮立体面板，深度展示该官职的维度数据。**点击面板以外的空白区域或再次点击原卡片可将其关闭。**

### 2.4 四维文化数据展示 (Four-Dimensional Cultural Lore)
根据样图 `detail-panel-sample.png` 指引，悬浮详情卡采用流式垂直布局：
1. **内容显示 (Header)**: 
   - 展示大字号宋体“官职名称”，紧邻右侧展示中字号宋体“品阶”。
   - 品阶下方展示灰度英文翻译 (Italicized Serif)。
2. **职能详情 (Summary)**: 
   - 样式：深灰色黑体，行高 1.6。内容展示职责原文。
3. **官职专属插图 (Illustration)**:
   - 视觉：低饱和度水墨化，边缘柔和羽化渐隐，完美融入宣纸背景。
4. **品秩与岁廪 (Salary/Stipend)**:
   - **样式**: 采用横向流式布局，直接浮于背景上。
   - **元素**: 每个属性由一个 **朱红小方印** (CSS 2px border) 与 **宋体数值** 组成，取消外部容器。
    - **交互**: 
      - **PC 端**: 悬停 (Hover) 触发沉浸式 Tooltip。
      - **移动端 (Touch)**: 点击 (Tap) 小方印或数值唤起 Tooltip，再次点击或点击外侧空白区域关闭。
    - **Tooltip 样式 (V3)**: 
      - **定位**: 采用 Portal 渲染。支持 **垂直翻转**（高度检测）与 **水平避让**（吸附边缘 10px）。
      - **文案结构**: 采用“历史原貌 + 趣味科普”双层结构。科普区由 `[💡 ...]` 标识，自动应用灰度斜体并带顶边线。
5. **典故连珠 (Lore Pearls)**: 
   - 布局：左侧放置“毛笔/卷轴”融合图标，右侧为模块标题“典故连珠”。
   - 展现：采用大字号、沉浸式宋体展示诗词/名言。右下角点缀一枚极小红印。
   - 展开：超过 3 行（或 60 字符）自动出现渐变遮罩与“更多/收起”按钮，支持内联展开阅读全文。
6. **风流人物 (Figures)**: 
   - 布局：左侧放置 `Users` 组员图标，右侧为模块标题“风流人物”。
   - 排版：小字号、灰度。横向流式布局，名字间以顿号分隔。
7. **互动入口 (CTA)**: 
    - 底部横向撑满的朱砂红 (#af292e) 按钮，文案为 **“接旨赴任”**。
    - **核心逻辑**: 点击后直接进入入仕模拟器（Simulator），透传当前官职数据进行“短路”匹配（绕过现代职业输入），直接生成海报，缩短转化链路。

### 2.7 详情面板交互与边缘情况 (Detail Panel UX & Edge Cases)
- **动态显隐**: 若无“典故”或“人物”，相关模块隐藏，插图与 CTA 按钮平滑衔接。
- **防止穿透**: 阻止点击/滚动事件冒泡至底层架构树。
- **出场动画**: 300ms ease-out 右侧滑入 + 渐现。

### 2.5 全局检索与典故中心 (Global Search & Lore Center)
1. **功能定位**: 打破单一组织架构的枯燥感，作为贯穿全站的“文化虫洞”。实现“人-官-文”的深度联动，允许用户跨越层级直接穿透定位到目标官职。
2. **入口与形态**: 功能深度集成于全局检索面板中。用户通过点击侧边栏顶部的“搜索”图标触发检索面板，内容涵盖所有官职、人物及典故。原独立的“典故连珠”标签页已根据需求移除，以保持导航栏精简。
3. **多维模糊检索**:支持按 官职名（如“司马”）、历史人物（如“白居易”）、诗词典故（如“江州司马”）三个维度进行跨朝代搜索。**搜索逻辑需忽略关键词及被搜索内容中的空格（如输入“吏部”应能匹配“吏  部”）。**
4. **结果展示**:搜索结果以精简列表呈现，包含：官职名称、品阶、简短的职权白话翻译  以及匹配到的高亮命中词。
5. **全局穿透与自动寻路 (Pan, Zoom & Auto-Expand)**:
  - 祖先反查与排他展开: 用户点击检索结果后，系统需通过数据字典向上反查出对应的一条直达链路，无视当前折叠状态强制展开这些祖先节点及其连线挂载到 DOM 中，同时自动折叠无关分支（Accordion 模式）。
  - 视界平滑定位: 随其布局渲染完毕，主界面的“官阶巡礼”树状图需自动平滑移动（Pan）并缩放（Zoom），将目标官职节点居中显示。
  - 状态联动: 目标节点自动触发边缘发光高亮选中态，并在左上角“云迹”面包屑中更新。
  - 详情呼出: 定位完成后，右侧自动弹出悬浮详情卡片，直接向用户展示该官职完整的四维文化数据（含历史配图、人物及引言典故）。

#### 2.6 “入仕”模拟器与裂变引擎 (命运卷轴流转逻辑)

* **功能描述:** 趣味互动模块，打破古今壁垒。用户输入现代职业，系统匹配贴合的古代官职，增强社交传播属性。整个交互流程必须是“输入 -> 宣判”的单点结果，严禁展示搜索列表，以保证“确权”的仪式感。
* **交互流程与细节要求:**
  1. **极简入场:** 界面仅保留极简输入框让用户输入现代职业。微交互提示文案：“你的好友已入局大唐官场，你呢？”（需能从全局变量中动态替换朝代名称，如果是通用版本则显示“古代官场”）。
  2. **触发动作 (CTA):** 采用底部横排撑满的按钮，背景色强制为朱砂红 (`#af292e`)，文案为“启封命运卷轴”。
  3. **匹配黑盒逻辑:**
     - **字典预设**: 前端进行智能拦截（如：“程序员”->“将作监丞”）。如果有“一对多”情况（如“产品经理”映射到“中书舍人”与“考功司员外郎”），则通过 `Math.random()` 随机抽取唯一结果展示。
     - **搜索兜底**: 对于字典外词汇，静默调用 Meilisearch 引擎获取相关度最高结果兜底。
  4. **海报材质与生成:** 动态生成带新中式边框的“委任状”海报。其背景必须叠加宣纸暖白底色 (`#f5f5dc`) 以及木浆纤维噪点。系统全局背景采用 `rushi_bg.png` 高保真纹理以增强沉浸感。系统全程必须基于纯前端 `html2canvas` 合成实现服务器零渲染压力，支持长按或分享。
     - **官名与品阶排版**: 采用竖排右起模式。主官名居右，官品居左。移除中间红点，由 32px 留白实现视觉隔离。
     - **降格设计**: 官品顶部对齐主官名后下沉 60px，形成具有律动感的视觉层次。
     - **官品装裱样式**: 升级为双线装裱印章风格。设置 `border: 2px solid #af292e`（外粗），并配合 `outline: 1px solid #af292e; outline-offset: -5px`（内细），背景保持完全透明。
  5. **盖印动效 (The Stamp Ritual):** 海报生成的瞬间，播放一枚专属朱红印章的动效。此步骤需运用严格的物理砸击 `@keyframes stampSmash`，让印章从高处（Scale 4）重力砸下，在 60% 进度时产生挤压反馈（Scale 0.9），随后进行轻微反弹（Scale 1.05）并最终定型。撞击瞬间配合全卡片的 `posterVibrate` 震动反馈，强化“砸击”质感。其背景图采用 `yinzhang.png` 高保真材质，并在其中心动态叠加热渲染每一个职业的专属印文。
     - **印章排版标准**: 采用 2x2 绝对定位矩阵。右上为“命”，右下为“数”，左上为“堪”，左下为“合”。
     - **物理参数**: 容器 90x90px，文字区域采用绝对定位解耦。
       - 堪/命: `top: 5px`, `LR: 12px`, `38px/900`
       - 合/数: `bottom: 17px`, `LR: 12px`, `38px/900`
     - **混合模式**: 强制使用 `mix-blend-mode: multiply` 实现入木三分的透墨质感。
  6. **裂变锚点:** 海报右下角置入专属水墨风二维码，并内置 UTM 参数强制重定向到入仕模拟器主站落地页。
  7. **800ms 延迟引流 (The 800ms Rule):** 在卡片完全展示及盖印仪式结束以后，由前台强制延迟 800ms 时长才能 Fade-in 底部的“造办处”关联文创入口卡片，若用户点击则将他们向展馆平滑过渡或触发302导流。

#### 2.7 商业引流与“造办处”文创展览 (Cultural Merch Exhibition - "The Imperial Workshop") 

* **功能定位:** 在不破坏“官阶巡礼”纯粹科普体验的前提下，建立高转化率的商业引流阵地。平台自身不处理订单与交易，仅作为“高品质文化展览与引流漏斗”。
* **入口设计:**
  * **全局常驻:** 置于左侧深色导航侧边栏 (Sidebar Navigation) 的常驻功能键，命名为“造办处”或“珍宝阁”，图标采用极简的折扇或锦囊融合设计。
  * **场景触发:** 由“入仕模拟器”结果页或特定的“典故悬浮卡片”底部关联顺滑呼出。
* **页面形态 (无头电商展示):**
  * 延续新中式极简风 (Neo-Chinese Minimalist) 与“水墨卷轴”结合的整体风格 ，采用“线上美术馆”的画廊式网格布局展现文创商品。
  * 剥离传统的“加入购物车”概念，以“赏鉴”、“请回案头”等古风文案作为 Call-to-Action (CTA) 按钮。
  * **商品按热度排序与红心点赞**: 商品卡片需展示“喜欢”按钮（如印章/红心），支持用户点赞并实装微动效，防止本地重复刷赞。商品列表强制按喜欢数量 (`likes_count`) 降序排列。
  * 图片规格：宣传图片规格推荐采用 800 × 600 像素或 1200 × 900 像素的图片素材。
* **导流机制 (重定向追踪):**
  * 用户点击意向商品后，系统需先触发埋点事件（记录来源页 ID、商品 ID），随后通过 302 重定向或拉起外部小程序，无缝跳转至指定的第三方电商平台（淘宝/微店/小红书）。

## 3. 架构与数据模型

### 3.1 核心技术栈与架构

* **Frontend (前端):** React 18, Vite.
* **Styling (样式):** TailwindCSS (Custom configuration for Neo-Chinese theme variables).
* **Icons (图标):** Font Awesome 6 (React components) or SVG.
* **Fonts (字体):** Noto Serif SC, Inter (Google Fonts).

* **数据架构 (全静态化 - Serverless):**
  * ~~Backend & API: Directus v11.16~~ → **已砍掉。** 所有官职、典故、俸禄数据直接硬编码为前端 JSON 文件 (src/data/dynastyData.js, src/data/products.js)，随 Vite 打包分发，零后端依赖。
  * ~~Database: PostgreSQL + ltree~~ → **已砍掉。** 数据已从数据库全量导出并固化为静态 ES Module。
  * ~~Search: Meilisearch~~ → **已砍掉。** 改为纯前端客户端模糊搜索 (src/api/localData.js: searchNodes())。
  * **恢复路径:** 原 Directus API 客户端 (src/api/directus.js) 保留未删除，恢复数据库模式只需将各组件 import 从 localData 切回 directus 即可。

* **核心渲染与动效引擎规范 (Interaction & Animation Engine):**
  * **零延迟热更机制 (Zero-Latency SVG DOM Pathing)**: 针对树状结构的 SVG 网状连线重绘，绕过 React 的 `setState` 批处理延迟，采用底层 `requestAnimationFrame` 配合 `MutationObserver`，在监听到子树展开时，直接执行高频 DOM 原生提权操作 (`setAttribute('d')`)，将 1 帧刷新差导致的“连线拉伸回弹（Rubber-Banding）”现象从物理根源上彻底抹除，提供紧密咬合的主机级 UI 帧同步感。
  * **物理遮罩与动态测距 (Dynamic Layout Width & Mask Constraint)**: 废弃 CSS 中 `maxWidth: 5000px` 等粗暴动画设置。动画展开时必须通过底层的 `scrollWidth` 或者 `ulReact` 进行真实物理包围盒读取，以真实计算出的节点尺寸作为 CSS `transition` 插值的目标以保证 800ms 过渡匀速跑完。同时利用这些边界值强制约束 SVG 最远连线断点，实现连线像卷轴一样“不跨越面纱外层”的物理剥离感。
  * **抗布局偏移追踪 (View Camera Pan-Lock)**: 任何子节点的横向撑开动画必须被父级的视角相机通过反向运算实时抵消，使得被操作（点击）的节点在屏幕视觉中纹丝不动，其他兄弟节点呈现完美的排斥退让位移。
  * **交错脱钩入场 (Decoupled Staggered Drop)**: 容器的生长动画需与内部卡片的展示脱钩，容器先行平滑生长，内部卡片利用 `transitionDelay` (例如 `index * 60ms`) 并在入场前强制 `translateY(-20px) scale(0.95)` 实现错位下落。其产生的卡片顶部锚点物理位移亦可同步拖动底部 SVG 连线同步下落。

* **Infrastructure (基础设施):** 强依赖 CDN (内容分发网络) 。所有高保真背景图、印泥动效图、水墨遮罩等静态资源必须全量上 CDN，确保即便裂变拉新导致高并发，主站依然能丝滑加载 。

* **用户追踪与埋点 (Tracking Analytics):**
  * **无感匿名追踪 (Frictionless Anonymous Tracking):** 鉴于 MVP 阶段坚决不引入繁杂的用户注册登录系统，前端需利用 `LocalStorage` 或轻量级浏览器指纹技术在端侧生成匿名 UUID 。此 UUID 将作为收集用户行为埋点（如模拟器裂变转化率、商品点击穿透率）时的唯一身份标识发送至后端接口 。以此在保持极致顺滑体验的同时，实现精准的商业引流漏斗分析。

* **Architecture Highlights (架构亮点):**
  * **视觉-逻辑解耦 (Visual-Logical Decoupling):** 引入稳定的 CardContainer 容器层 。树状连线的坐标计算仅依赖该静态容器，而视觉动效（如 Hover 上浮、选中缩放）仅在容器内部发生，确保交互过程中连线的绝对稳定性 。
  * **递归可见性检查 (Recursive Visibility):** 树状连线渲染器会递归检查祖先节点的展开状态，确保折叠子树内部的连线被物理隐藏，避免“虚影线”残留 。
  * **动静分离与防雪崩:** 核心科普（静态缓存优先）与埋点重定向（异步处理）在接口层严格分离，确保引流模块的高并发不会拖垮主站 。

* **Structure (结构):** Component-based architecture 。 State managed via React hooks 。 `data.js` 及其内容将完全迁移至 PostgreSQL 数据库管理 。
* **Key Components (核心组件):**
  * `HierarchyTree`: 交互式官僚组织架构树。
  * `DetailPanel`: 双层信息架构的侧边详情面板。
  * `SalaryFlowBoard`: 高性能、带科普 Tooltip 的流式俸禄展示组件。
  * `AppointmentPoster`: 入仕模拟器核心海报生成组件。
* **Tools (工具):** `html2canvas` (或类似库) 用于海报生成模块 。

### 3.2 数据结构规范 (Data Model)
所有的朝代数据需遵循严格的 JSON schema `node`：

```typescript
interface Node {
    id: string;
    title: string;        // 官职名称 (如: 中书省)
    type?: string;        // 官职类别标记 (如: admin, military, finance) => 映射小图标
    level: string;        // 官阶 (如: 正二品)
    salary?: Record<string, string>; // 俸禄数据。Key 由朝代特定体系决定 (见下文映射表)，Value 为数值或文本。
    description: string;  // 详细/摘要职责
    figures?: string[];   // 代表人物数组
    allusions?: Array<{ title: string, text: string }>;  // 典故数组
    poetry?: Array<{ author: string, poem: string }>;    // 诗词数组
    children?: Node[];    // 下属机构/官职
}
```

### 3.3 运行与部署 (Running & Deployment)

* **一键启动 (推荐)**: 在根目录下运行 `start.bat` 或执行 `npm start` / `npm run dev:all`。这将开启控制面板并由脚本自动拉起前端并启动后端 Directus 服务。
* **停止服务**: 在根目录下运行 `stop.bat` 或执行 `npm run stop`。脚本将强制释放 5173 (Vite) 与 8055 (Directus) 端口。
* **手动启动**:
  * **前端**: 根目录下运行 `npm run dev` (Vite, 默认 5173 端口)。
  * **后端**: `backend` 目录下运行 `npm start` 或 `npx directus start` (Directus, 默认 8055 端口)。
* **环境要求**: Node.js 18+, PostgreSQL (由后端配置连接)。

## 4. 图像资产规范 (Image Asset Convention) - 【AGENT 执行准则】

### 4.1 详情面板资产要求
- **宣纸底纹 (Paper Texture)**: 
  - **路径**: `/public/assets/ui/paper-texture.png` (或复用 `main-bg.png`)。
  - **渲染**: 必须平铺且无缝，确保在不同高度的卡片下背景质感一致。
- **水墨羽化遮罩 (Ink Mask)**:
  - **路径**: `/public/assets/ui/ink-mask-large.png`。
  - **效果**: 必须是 8-bit Alpha 通道的 PNG，边缘呈现不规则的水墨飞白，用于卡片边缘和插图边缘的渐隐。
- **官职插图 (Illustrations)**:
  - **路径**: `/public/assets/content/[role_name].png`。
  - **风格**: 必须保持**低饱和度、水墨渲染**感。
  - **处理**: 严禁拉伸，强制使用 `object-fit: contain`。必须配合 `-webkit-mask-image` 实现边缘羽化，禁止出现直角边框。

### 4.2 存储与读取规则
- **严禁硬编码路径**: 详情面板的背景图、遮罩图需引用 CSS 变量或全局常量。
- **动态读取**: 业务插图（Figure/Illustration）必须从 `data.js` 的 `panelImage` 字段读取，严禁在组件内直接 `import` 业务图片。
- **缺省处理**: 若字段为空，必须回退至 `/public/assets/content/horse_rider.png`。

### 4.3 社交引流交互流转图 (User Flow)：命运卷轴 (The Scroll of Destiny) 转化漏斗
1. **[站外触达] 扫码入局**
   * **场景:** 用户 B 在社交平台（如朋友圈）看到用户 A 分享的“入仕委任状”海报。
   * **动作:** 用户 B 扫描海报专属的水墨风二维码，进入带有 UTM 追踪参数的链接（如 `qingyunzhi.com/simulator?ref=poster_share`）。
2. **[站内承接] 极简入场**
   * **场景:** 微信等内置浏览器直接平滑拉起网页，进入主站“入仕模拟器”模块。
   * **动作:** 界面展现极简输入框及微交互文案（如：“你的好友已入局大唐官场，你呢？”）。
3. **[核心体验] 职业匹配**
   * **动作:** 用户 B 输入现代职业，点击生成。系统智能输出匹配的古代官职。
4. **[情绪高点] 委任状生成**
   * **场景:** 伴随“盖印”动效，生成专属用户 B 的新中式委任状海报。
5. **[顺水推舟] 雅玩引流**
   * **动作:** 海报下方延迟淡入“造办处”文创推荐卡片（如：“履新此职，怎能少了一把好扇？ 👉 赴【造办处】赏玩同款水墨折扇”）。用户点击后重定向至外部电商完成最终转化。
